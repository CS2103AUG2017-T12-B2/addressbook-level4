Enhancement Added: Sending email with a logged in email

External behavior

Start of Extract [from: User Guide]

Sending an email to a list of recipients: email_send

Sends an email to a list of recipients. Requires an logged in email using email_login
Format: email_send "[RECIPIENTS]" "[TITLE]" "[BODY]"

  • "[RECIPIENTS]" is the list of recipients, separated by ;. Example: "cs2103testacc@gmail.com;cs2103test@yahoo.com"

  • "[TITLE]" is the title of the email. Example: "Change of class schedule"

  • "[BODY]" is the body of the message. Example: "The class schedule is changed to 4 pm"

Example:

  • email_send "cs2103testacc@gmail.com" "Test Title" "Test Body"
    send an email with given title and body to a single recipient.

  • email_send "cs2103testacc@gmail.com;cs2103testacc@yahoo.com" "Test Title" "Test Body"
    send an email with given title and body to a list of recipients.

End of Extract

Justification

As the targeted users are tutors and businessmen, email functionality is required for information dissemination.

Enhancement Added: Find person(s) without matching the whole keyword

External behavior

Start of Extract [from: User Guide]

Find a person without matching the whole keyword: find_contain

Finds all persons whose name / phone / address / email contains the given string
Format: find_contain PREFIX_PERSON_ATTRIBUTE/STRING [MORE_PARAMETERS]…​

  • The search is an "AND" search.

  • The characters that follow a prefix are treated as one string.

  • A match happens when a person’s details contain the given strings e.g. "David Li" contains "avid", "93463991" contains "99".

End of Extract

Justification

  • The find command is an OR search, therefore if users want to filter out unwanted contacts by giving more information, an AND search (find_contain) will be required.

  • For the search bar UI enhancement, search without matching the whole keyword is more desirable (for example: typing ale will immediately show all Alex, thus saving time.

Developer Guide

Start of Extract [from: Developer Guide]

Search Bar implementation

The Search Bar is a GUI feature that allows users to type into 5 text fields (for name / phone / address / email / tag) and filters those whose details contain the string in the respective text fields.

Every text field in the Search Bar listens to changes and calls onSearchBarChanged if an input is detected. The onSearchBarChanged function then collates information in all 5 text fields and calls a find_contain command with the collated information.

Sequence diagram of Search Bar

  • SearchBar.java inherits from UIPart and therefore using FXML a function onSearchBarChanged() that is called every time a text field changes can be attached to each of the search field.

  • onSearchBarChanged() reads current texts in all search fields and builds a command input for find_contain.

  • The command input is then passed to LogicManager and AddressBookParser then FindContainCommandParser respectively.

  • As such, a new find_contain command is executed every time there is a change in the search bar. The find_contain passes a new predicate to the ModelManager to update the filtered person list.

Design Considerations

Aspect: Implementation of SearchBar.java
Alternative 1 (current choice): Add a new SearchBar that inherits UIPart
Pros: Very easy to find and filter desired contacts
Cons: Additional data structures that might cost more space.
Alternative 2: Just use the find_contain command
Pros: Uses entirely existing methods to implement.
Cons: Slow and time consuming for users.

End of Extract

Justification

  • Sometimes users need to quickly find a person and the search bar provides a quicker way than command line.

  • Users sometimes forget contacts' details: for example, users call a person "Dave" but do not remember whether he saved the contact as "David" or "Dave". Search Bar solves this problem by allowing users to just type "dav" in the name field rather than use the find command twice for "dave" and "david".

Start of Extract [from: Developer Guide]

Email Component

The email component ultilised the external library javax.mail for sending and receiving emails.

The EmailManager.java class implementing Email manages the login status by keeping an Autheticator object along with its email String.

As the library javax.mail uses IMAP protocol for receving emails and SMTP for sending emails, the EmailManager also holds a Hashtable of IMAP hosts and SMTP hosts / ports for different email providers. This allows future support for more providers.

Design Considerations

Aspect: Implementation of Email
Alternative 1 (current choice): Create a new Email Component
Pros: Can be further improved to work with other commands to quickly send emails to a group of people
Cons: Command line interface is not user friendly
Alternative 2: Use the built in browser to send emails
Pros: Already has the GUI which is user friendly, support for all email providers
Cons: However, it will be more difficult to work with other commands to send to a specific groups of contacts.

End of Extract

To the left of every contact card, there is a check box. When the email send window opens, the recipient field will automatically contain all ticked persons' emails.

The ticked person cards’s references are stored in an ArrayList in PersonListPanel.java. The following sequence happens when a person card is ticked.

Also, a listener is added to the ObservableList containing the filtered persons to show in the panel. When a change in the ObservableList is listened, the TickedPersons ArrayList is reset.

personList.addListener(new ListChangeListener<ReadOnlyPerson>() {
            @Override
            public void onChanged(Change<? extends ReadOnlyPerson> c) {
                tickedPersons = new ArrayList<PersonCard>();
            }
        });
